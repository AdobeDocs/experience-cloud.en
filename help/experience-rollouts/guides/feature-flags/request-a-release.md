# Request a release {#request-a-release}

## Prerequisites {#prerequisites}

* You have the **Release Manager** role — see [User roles](../teams/user-roles.md)
* Your application is onboarded — see [Onboard your application](../applications/onboard-your-application.md)

>[!TIP]
>
>Before requesting a release, review [Releases and cross-team feature groups](releases-and-cross-team-feature-groups.md). A cross-team feature group may meet your needs with less overhead — it is self-serve and supports richer audience targeting.

## Submit a support request {#submit}

Release creation is not self-serve. Contact Experience Rollouts support to request a new release. Provide the following information:

| Field | Description |
|---|---|
| **Name** | A unique name for the release. Do not include spaces in the release name. |
| **Team** | The team that will own the release. |
| **Environment** | The environment where the release should be created (Stage or Production). |
| **Objective** | A brief description of what this release will roll out and why. |
| **Applications** | The application or applications to include in the release. |
| **Duration** | How long you need the release to remain active. Releases are expected to be closed or baselined within three months. If you need longer, provide a business justification. |

>[!NOTE]
>
>The number of active releases at any given time is limited. Plan to baseline or close your release within three months to free up capacity for other teams.

## After the release is created {#after}

Once you receive confirmation that the release has been created, log in to the console and complete the release configuration. See [Release workflow end-to-end](release-workflow-end-to-end.md) for the full sequence of steps.
